# 智手®（ZhiHand）路线图

说明：智手®是 ZhiHand 的中文名称；ZhiHand 由 HandGPT 更名而来。文档中的域名、包名、命令与代码标识保持英文。

这份路线图描述的是公共核心仓库视角下的交付规划。

它有意聚焦于共享契约和公共集成边界，而不是按名称列出私有实现仓库。

## Phase 1：共享控制模型

目标：让共享控制模型可以在多个宿主环境和运行时类别中真正执行并复用。

主要结果：

- `control.proto` 稳定到足以支撑真实集成
- `zhihandd` 提供可用的参考控制面
- 首个公共宿主适配器可以接入同一套模型
- 共享模型可以扩展到更多宿主环境

关键工作：

1. 完成第一版控制域消息集
2. 在 `zhihandd` 中实现服务骨架和请求路由
3. 定义 streaming、生命周期、错误和 capability 语义
4. 通过首个公共宿主适配器打通一条端到端集成路径
5. 发布针对宿主适配器、移动端、设备端和 Web 运行时的集成预期

退出标准：

- 核心协议消息已经版本化并有文档
- `zhihandd` 能接收并路由至少一条完整控制流程
- 至少一个宿主环境已经基于同一协议完成可验证集成

## Phase 1.5：部署边界

目标：定义清晰的公共 / 私有边界，使部署专属控制面可以存在，同时不污染公共核心。

主要结果：

- 公共模型中用于配对签发的边界已经明确
- 公共模型中用于 endpoint 命名和 provisioning 的边界已经明确
- 公共模型保持对私有基础设施实现中立

关键工作：

1. 定义公共模型暴露的 pairing session 边界
2. 定义宿主适配器与部署控制面之间的 provisioning 边界
3. 明确哪些能力属于公共核心，哪些能力属于私有基础设施
4. 为未来中继与审计能力预留边界

## Phase 2：运行时类别集成

目标：让公共模型可被主要运行时类别一致使用。

主要结果：

- 宿主适配器可一致接入
- 移动 App 可一致接入
- 设备运行时可一致接入
- Web 运行时在适当边界内可一致接入

关键工作：

1. 定义 capability 协商与版本兼容规则
2. 在公共契约层建立认证、身份和信任边界
3. 记录面向设备侧的传输假设
4. 围绕协议兼容性建立共享测试或测试夹具

## Phase 3：产品化

目标：从联通原型走向长期稳定演进的公共核心。

主要结果：

- 公共核心具备稳定的发布流程
- 协议与适配器之间具有清晰的兼容性策略
- 可观测性、失败处理和迁移指导更加完善

关键工作：

1. 发布版本与兼容性策略
2. 为 `zhihandd` 提供部署与打包指导
3. 形成公共日志、追踪和恢复预期
4. 编写旧集成的迁移指南

## 待决问题

- 有多少宿主特有行为应该保留在适配器中，而不是放入共享服务
- 公共模型应在多大程度上描述部署控制面的交互边界
- 除了 `zhihandd` 之外，硬件通信是否还需要独立的公共传输服务

## 当前优先级

1. `control.proto`
2. `zhihandd`
3. 公共宿主适配层边界
4. 首个宿主适配器集成
5. 运行时类别集成契约
